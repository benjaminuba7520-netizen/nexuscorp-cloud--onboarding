NexusCorp Global — Infrastructure Engineering Onboarding

The goal of this project is to document real-world cloud engineering deliverables across security, compute, storage, and deployment automation.

INTRODUCTION

During an internal security audit at NexusCorp, it was discovered that contract developers were sharing root account credentials to test applications — a major security risk that exposed the organisation to financial loss, data breaches, and compliance violations.

My job was to fix that. This article documents how I achieved the following security milestones step by step:

  • Hardened the root account by configuring virtual Multi-Factor Authentication (MFA) using an authenticator app.
  • Isolated financial monitoring by creating a Finance-Auditors IAM User Group with the AWS Billing managed policy attached.
  • Established an Admin Proxy user with administrator console permissions, removing all reliance on the root account for daily operations.
  • Implemented Developer RBAC by creating an Engineering-ReadOnly-Devs IAM User Group with the ReadOnlyAccess managed policy.
  • Provisioned a restricted sandbox identity for junior developer Alex, limiting access to read-only visibility with no ability to modify production environments.
  • Verified the access restrictions by logging in as alex-dev and confirming that all attempts to create resources were blocked by AWS.

Follow along to see exactly how each milestone was achieved, with screenshots at every step.

The Business Problem

Contract developers were sharing root account credentials to test applications. This created shared credential risks, triggered audit findings, and exposed NexusCorp to serious compliance violations.

My Objective

As the engineer assigned to this ticket, I was tasked to achieve the following:

  1. Harden the Root Identity — Log into the AWS Root account and immediately configure virtual MFA using an authenticator app.
  2. Isolate Financial Monitoring — Create an IAM User Group named Finance-Auditors and attach the AWS-managed Billing policy.
  3. Establish an Admin Proxy — Create a personalised IAM User, assign it to the Finance-Auditors group, and grant administrator console permissions.
  4. Implement Developer RBAC — Create an IAM User Group named Engineering-ReadOnly-Devs and attach the ReadOnlyAccess managed policy.
  5. Provision the Developer Identity — Create an IAM User named alex-dev and add them to the Engineering-ReadOnly-Devs group.
  6. Verify Access Restrictions — Log in as alex-dev and confirm the access block is working correctly.

STEP 1 — SECURING THE ROOT ACCOUNT

1.1 — Console Sign In

Navigate to the AWS Console and select sign in as root user.

1.2 — Root Account Sign In

Enter the root account email address to proceed with login.

1.3 — Email Address Input

Input the registered root email address associated with the AWS account.

STEP 2 — ENABLING MFA ON THE ROOT ACCOUNT

2.1 — Search for MFA Settings

In the AWS Console, search for MFA to locate the security credentials settings.

2.2 — Add MFA for Root User

Select the option to assign an MFA device to the root user account.

2.3 — Device Name & Authenticator App

Enter a device name and select Authenticator App as the MFA method.

2.4 — Authenticator App Setup

Open your authenticator app and prepare to scan the QR code.

2.5 — Barcode Scan

Scan the QR barcode using your authenticator app to link the MFA device.

2.6 — MFA Successfully Assigned

MFA has been successfully assigned to the root account. The root account is now protected with two-factor authentication.

STEP 3 — ISOLATE FINANCIAL MONITORING

3.1 — Search for IAM

In the AWS Console search bar, type IAM and select the IAM service to begin setting up user groups.

3.2 — Click on IAM User Groups

Inside the IAM Console, navigate to User Groups and click Create Group.

3.3 — Choose Group Name

Name the group Finance-Auditors to clearly identify its purpose and scope.

3.4 — Select AWS Managed Policy

Search for and select the AWS managed policy to attach to the Finance-Auditors group.

3.5 — Attach Billing Policy

Attach the Billing managed policy to the Finance-Auditors group, granting financial visibility without admin access.

STEP 4 — ESTABLISH YOUR ADMIN PROXY

Create a personalised IAM User for yourself, assign it to the Finance-Auditors group, and grant it administrator console permissions. Once done, log out of the root account immediately — from this point forward, the root account is off-limits.

4.1 — Navigate to IAM Users

Navigate to your search bar and search for IAM.

4.2 — Create New IAM User

Click IAM User and begin creating your personalised admin user.

4.3 — Set Console Access

Enter your personalised username, enable AWS Management Console access and set an initial password.

4.4 — Assign to Finance-Auditors Group

Add the new user to the Finance-Auditors group to inherit its billing permissions.

4.5 — Grant Administrator Permissions

Attach the AdministratorAccess policy directly to grant full console permissions.

4.6 — Admin User Successfully Created

The admin proxy user has been created successfully. AWS will now present you with two options to retrieve the login credentials:

  • Email sign-in instructions — AWS sends the console login URL and username directly to the user's email address. This is useful when onboarding a team member remotely.
  • Download the CSV file — AWS generates a CSV file containing the console login URL, username, and temporary password. Save this file securely as it will not be available again once you leave this page.

Once credentials are saved, log out of the root account immediately. From this point forward, all AWS activity must be performed through this admin IAM user — the root account is now off-limits.

STEP 5 — IMPLEMENT DEVELOPER RBAC

With the root account locked down and the admin proxy established, the next phase involved logging in as the Benjamin IAM user, activating billing access, and implementing Role-Based Access Control (RBAC) for the development team.

5.1 — Log In as Benjamin IAM User

Log out of the root account and sign in using the newly created Benjamin IAM admin user credentials.

5.2 — Benjamin IAM User Console Access Confirmed

Successfully logged in as the Benjamin IAM user. The root account is no longer in use.

5.3 — Activate IAM User and Role Access to Billing

Navigate to the Billing settings and activate IAM user and role access to allow the Finance-Auditors group to view billing information.

5.4 — Billing Access Successfully Activated

IAM user and role access to billing has been successfully activated, allowing the Finance-Auditors group to monitor financial activity.

5.5 — Create Engineering-ReadOnly-Devs Group

Navigate to IAM User Groups and create a new group named Engineering-ReadOnly-Devs to manage developer access.

5.6 — Attach ReadOnlyAccess Policy

Attach the AWS managed ReadOnlyAccess policy to the Engineering-ReadOnly-Devs group, ensuring developers can view but not modify any resources.

5.7 — Confirm Group and Policy Configuration

Review and confirm the group settings before finalising the Engineering-ReadOnly-Devs group creation.

5.8 — Developer RBAC Successfully Implemented

The Engineering-ReadOnly-Devs group has been successfully created with the ReadOnlyAccess policy attached. Developer RBAC is now active.

STEP 6 — PROVISION THE IDENTITY (ALEX-DEV)

With the Engineering-ReadOnly-Devs group in place, the final step was to create an IAM User named alex-dev, generate an initial console password, and add them to the Engineering-ReadOnly-Devs group — restricting their access to read-only visibility with no ability to modify production environments.

6.1 — Navigate to IAM Users

Inside the IAM Console, navigate to Users and click Create User to begin provisioning the alex-dev identity.

6.2 — Create Alex-Dev User

Enter alex-dev as the username and enable AWS Management Console access.

6.3 — Generate Initial Console Password

Set an initial console password for alex-dev and require a password reset on first login.

6.4 — Add to Engineering-ReadOnly-Devs Group

Assign alex-dev to the Engineering-ReadOnly-Devs group to inherit the ReadOnlyAccess policy.

6.5 — Review and Confirm User Settings

Review all the settings for alex-dev before finalising the user creation.

6.6 — Alex-Dev Successfully Created

The alex-dev user has been successfully created and added to the Engineering-ReadOnly-Devs group. Download the CSV file or send login credentials via email to complete the onboarding.

6.7 — Verify Alex-Dev Access

Log in as alex-dev to confirm read-only access is working correctly and production environments are fully restricted.
